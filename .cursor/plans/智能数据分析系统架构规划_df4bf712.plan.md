## 技术决策记录 (ADR)

### ADR-001: LangChain 接入 Qwen3 的方式
- **决策**：采用 `langchain-openai` + 阿里云百炼 DashScope OpenAI 兼容接口（**方案 A**）
- **否决方案**：`langchain-community` 中的 `ChatTongyi`（**方案 B**），已 deprecated，v1 路径下迁入 `langchain-classic`
- **依据**：
  1. `langchain-openai` 官方长期主推，0.x ~ 1.x 跨版本兼容
  2. 流式响应、工具调用、structured output 全功能支持
  3. 与 OpenAI SDK 同源，团队未来切换 OpenAI / DeepSeek / 其它 OpenAI 兼容服务零成本
  4. DashScope 提供 `/compatible-mode/v1` 端点，协议完全对齐
- **调研来源**：docs-langchain MCP 中 `chat/index.mdx` 仅列出 `Qwen QwQ Qwen` 索引占位，**未提供 `ChatTongyi` 独立页面**；选 A 是为降低未来依赖 `@langchain/classic` 的风险
- **状态**：✅ 已确认（2026-06-06），Phase 3 实施时按此落地
- **对应任务**：`p3-llm-qwen` / Linear NL2SQL-21

### ADR-002: Qwen3 (DashScope) 字段规范（实测 2026-06-06）
**实测来源**：`backend/test_qwen3.py`，使用 API key `sk-****`、模型 `qwen3-max`、base_url `https://dashscope.aliyuncs.com/compatible-mode/v1`。

#### 1. 基础调用（`openai.OpenAI(...).chat.completions.create()`）
| 字段路径 | 类型 | 实际值/说明 |
|---|---|---|
| `response.id` | str | `chatcmpl-{uuid}` |
| `response.model` | str | `qwen3-max` |
| `response.object` | str | `chat.completion` |
| `response.created` | int | Unix 秒（实测 `1780748948`） |
| `response.usage.prompt_tokens` | int | 实测 22 |
| `response.usage.completion_tokens` | int | 实测 32 |
| `response.usage.total_tokens` | int | 实测 54 |
| `response.usage.prompt_tokens_details.cached_tokens` | int | 实测 0（命中缓存为非 0） |
| `response.usage.completion_tokens_details` | obj | `None` |
| `response.moderation` | obj | `None` |
| `response.system_fingerprint` | str | `None` |
| `choices[0].finish_reason` | str | `stop`（正常生成）/ `tool_calls`（函数调用）/ `length`（超长截断）/ `content_filter`（敏感过滤） |
| `choices[0].index` | int | 0 |
| `choices[0].logprobs` | obj | `None` |
| `choices[0].message.role` | str | `assistant` |
| `choices[0].message.content` | str | 生成内容；**工具调用时为 `""`（空串）** |
| `choices[0].message.refusal` | obj | `None` |
| `choices[0].message.tool_calls` | list\|None | 见 §3 |
| `choices[0].message.function_call` | obj | `None`（已弃用字段，Qwen 走 `tool_calls`） |
| `choices[0].message.annotations` | obj | `None` |
| `choices[0].message.audio` | obj | `None` |

#### 2. 流式输出（`stream=True`）
- 返回类型：`ChatCompletionChunk`，与 OpenAI 协议一致
- 增量分词：中文为**单字级别**（`猫`、`对`、…）；UI 层若需"按词打字"必须自己合并
- **第一 chunk**：`delta.role = "assistant"`，`delta.content = ""`（仅元数据）
- **中间 chunk**：`delta.role = None`，`delta.content = "字"`
- **最后 chunk**：`finish_reason = "stop"`（或 `tool_calls`）
- ⚠️ **流式默认 `usage = None`**：要在最后一次统计 token，必须传 `stream_options={"include_usage": True}`

#### 3. 工具/函数调用（`tools=[...]`）
- `choices[0].finish_reason` = **`tool_calls`**（不是 `stop`）— 路由层必须按此分流
- `choices[0].message.content` = **`""`（空串）** — 不要把空内容当成错误
- `message.tool_calls` 是 `list`，**支持多 tool 并行调用**（实测一次返回 2 个）
- 每个 tool_call 结构：
  ```
  ChatCompletionMessageFunctionToolCall(
    id='call_77146d4c446a4e72b7303c21',          # 23 字符短 ID
    type='function',
    function=Function(
      name='get_weather',                       # str: 函数名
      arguments='{"city": "北京"}'              # ⚠️ JSON 字符串（不是 dict！）
    ),
    index=0                                     # 工具在 list 中的下标
  )
  ```
- ⚠️ **`function.arguments` 是 JSON 字符串**：业务侧必须 `json.loads()` 后再校验
- 回传工具结果的消息格式：
  ```python
  {"role": "tool", "tool_call_id": tc.id, "content": "...结果..."}
  ```
  回传时**必须带 `tool_call_id` 匹配**，否则 LLM 无法关联

#### 4. LangChain `ChatOpenAI` 封装字段映射
| OpenAI 原始 | LangChain 字段 |
|---|---|
| `choices[0].message.content` | `.content` |
| `choices[0].message.role` | `.type` = `'ai'`（固定值） |
| `usage.prompt_tokens` | `.usage_metadata.input_tokens` |
| `usage.completion_tokens` | `.usage_metadata.output_tokens` |
| `usage.total_tokens` | `.usage_metadata.total_tokens` |
| `usage.prompt_tokens_details.cached_tokens` | `.usage_metadata.input_token_details.cache_read` |
| `id` (chatcmpl-...) | `.response_metadata.id` |
| `model` | `.response_metadata.model_name` |
| `finish_reason` | `.response_metadata.finish_reason` |
| `model_provider` | `.response_metadata.model_provider` = `'openai'` |
| `choices[0].message.tool_calls` | `.tool_calls`（list of `ToolCall`） |
| 无效 tool_call | `.invalid_tool_calls` |
| `message.refusal` | `.additional_kwargs.refusal` |
| LangChain run id | `.id` = `lc_run--{uuid}` ⚠️ 与 OpenAI id 不同 |

- **流式返回** `AIMessageChunk`：`.tool_call_chunks` 增量信息，`.usage_metadata` 在流式下**通常为 None**（需在 `ChatOpenAI(stream_usage=True)` 显式开启）
- **最后 chunk** 的 `.response_metadata = {}`（流式 metadata 不一定在末尾）

#### 5. 实施铁律（Phase 3 强制遵守）
1. **路由层必须按 `finish_reason` 分流**：`stop` → 正常回复；`tool_calls` → 解析 `tool_calls` 列表；`length`/`content_filter` → 提示用户
2. **`function.arguments` 必须 `json.loads()`**：定义统一工具 `parse_tool_arguments(tc) -> dict`
3. **流式需要 token 统计时**：`openai` SDK 传 `stream_options={"include_usage": True}`；LangChain 端 `ChatOpenAI(stream_usage=True, ...)`
4. **多 tool 并行调用**：遍历 `tool_calls` 全部执行后再回传，不要只取第一个
5. **回传 tool 结果**：`role="tool"` + `tool_call_id` 必须精确匹配
6. **LangChain 取 token** 用 `.usage_metadata.input_tokens`，**不要**用 `.response_metadata.token_usage`（后者是旧版路径，未来版本可能移除）
7. **错误处理**：超时、网络、4xx、5xx 必须捕获；`finish_reason == "content_filter"` 时给用户友好提示而不是堆栈
8. **model_name 校验**：`response_metadata.model_name` 应等于预期模型，否则日志告警（可能路由错模型）

- **状态**：✅ 已实测落地（2026-06-06），Phase 3 编码必须遵循
- **对应任务**：`p3-llm-qwen` / Linear NL2SQL-21（description 已同步）
- **证据脚本**：`backend/test_qwen3.py`

### ADR-003: NL2SQL 实现路径（LangChain v1 `create_agent` 方案，2026-06-06 MCP 实测）
**调研来源**：`user-docs-langchain` MCP，`/oss/python/langchain/sql-agent.mdx` 完整章节（行 372-1013），是 LangChain **官方推荐**路径。

#### 1. 决策
采用 LangChain v1 的 **`create_agent()` + 4 个标准 SQL @tool** 模式实现 NL2SQL。
**否决** 旧版 `langchain-community` 的 `SQLDatabaseChain` / `create_sql_agent` —— 这两个 API 在 v1 主线**已迁入 `langchain-classic`**，新项目不要用。

#### 2. 核心组件（全部来自 MCP 文档原文）
```python
from langchain.tools import tool
from langchain.agents import create_agent
from langchain_openai import ChatOpenAI

# 4 个标准工具（来自 docs/sql-agent.mdx 行 372-440）
@tool
def sql_db_list_tables() -> str: ...
@tool
def sql_db_schema(table_names: str) -> str: ...
@tool
def sql_db_query(query: str) -> str: ...
@tool
def sql_db_query_checker(query: str) -> str: ...

# LLM（与 ADR-001 一致：Qwen3 via DashScope 兼容模式）
model = ChatOpenAI(model="qwen3-max", base_url="...", api_key=...)

# Agent 工厂
agent = create_agent(
    model,
    tools=[sql_db_list_tables, sql_db_schema, sql_db_query, sql_db_query_checker],
    system_prompt="...",   # 强烈建议显式注入业务 schema 与规则
)
```

#### 3. 工作流（agent 内部自动 ReAct 循环）
1. `sql_db_list_tables` — 列出所有表
2. `sql_db_schema` — 取相关表结构 + 样例行
3. 生成 SQL → `sql_db_query_checker` 自检
4. `sql_db_query` 执行
5. 出错 → 错误回灌 LLM → 重试（直到成功或达到最大步数）
6. 合成自然语言回答

#### 4. 关键设计要点
- **LLM 选**：必须支持 tool-calling（Qwen3 已在 `test_qwen3.py` 测试 3 验证 OK）
- **System Prompt**：必须包含"先 list_tables → schema → query_checker → query"的强约束，否则 LLM 可能跳步
- **HITL 强制开启**（docs/sql-agent.mdx 行 968）：在 `sql_db_query` 前用 LangChain 内置 HITL middleware 暂停，让用户/前端确认 SQL —— **生产环境必做**
- **多步可能**：一次自然语言查询可能触发 5-10 次 LLM 往返，token 消耗大，建议为 NL2SQL 调用单独预算
- **DB 权限**：官方原文强调"Use narrowly scoped database permissions"——给 agent 用的 SQLite 连接用单独用户，禁止 DROP/DELETE 等

#### 5. 实施铁律（Phase 3 `p3-nl2sql` 强制遵守）
1. **必须 `create_agent()`**，不要写 `SQLDatabaseChain`
2. **4 个 tool 都按官方名称**（`sql_db_list_tables` / `sql_db_schema` / `sql_db_query` / `sql_db_query_checker`），方便后续替换或共享
3. **`@tool` 装饰器的 docstring 必填且英文**，LLM 靠它判断何时调用
4. **system_prompt 必填**并包含：角色定位、可用表、列名提示、输出格式要求、查询步骤
5. **HITL 必须开启**，且拦截的是 `sql_db_query`（写操作可同时拦截）
6. **DB 连接单例化**，放到 `app/db/` 复用，不要每个 tool 重新建连接
7. **错误返回给 LLM 而非抛异常**（官方原文："error message is returned to the model"）
8. **生成 SQL 与执行 SQL 解耦**：`@tool` 内部只执行，前端可单独展示 SQL（用于 ADR-002 中的 `function.arguments` 字段展示）

#### 6. 与既有架构的衔接
- 工具列表 → 接入 Phase 3 `p3-nl2sql`（NL2SQL-24）
- `sql_db_list_tables` / `sql_db_schema` 的实现 → 复用 Phase 3 `p3-schema-api`（NL2SQL-26）
- HITL 中间件 → 接入 Phase 4 `p4-sse-stream`（NL2SQL-28，依赖流式暂停/恢复）

- **状态**：✅ 已 MCP 实测确认（2026-06-06）
- **对应任务**：`p3-nl2sql` / Linear NL2SQL-24
- **证据**：`/oss/python/langchain/sql-agent.mdx`（docs-langchain MCP 索引）

### ADR-004: NL2SQL Agent 字段规范（实测 2026-06-06）
**实测来源**：`backend/test_nl2sql.py`，使用 API key `sk-****`、模型 `qwen3-max`、langchain 1.3.4。端到端跑通"查询用户"问题，13.88s，5 次 LLM 往返。

#### 1. `agent.invoke()` 返回结构
```python
result = {"messages": [HumanMessage, AIMessage, ToolMessage, ..., AIMessage]}  # 仅 1 个顶层 key
# 最终回答：找到 type=='ai' 且无 tool_calls 的最后一条 AIMessage.content
```

#### 2. `AIMessage.tool_calls[i]`（⚠️ 与 OpenAI SDK 不同）
| 字段 | 类型 | 实测值 |
|---|---|---|
| `name` | str | `"sql_db_query"`（与工具注册名严格一致） |
| `id` | str | `"call_f049259036274ab3867de8d2"`（23 字符） |
| `args` | **dict** | `{"query": "SELECT ..."}` ⚠️ **LangChain 已 json.loads**，不需要再 parse |
| `type` | str | 固定 `"tool_call"` |

对比 ADR-002 中 OpenAI SDK 的 `function.arguments` 是 JSON 字符串 — **LangChain 层是 dict，业务侧不用再解析**。

#### 3. `ToolMessage` 字段
```python
ToolMessage(
    type='tool',
    name='sql_db_query',                                  # 工具名
    id='364d6af7-7afd-...',                              # ToolMessage 自己的 ID
    content="Columns: [...]\nRows:\n(...)",              # 字符串
    tool_call_id='call_f049259036274ab3867de8d2',         # ⚠️ 必须匹配 AIMessage.tool_calls[i].id
    status='success'                                      # 'success' | 'error'
)
```

**铁律**：`ToolMessage.tool_call_id == AIMessage.tool_calls[i].id` —— 不匹配 LLM 无法关联，会导致下次循环出错。

#### 4. `response_metadata.finish_reason` 流变（重要！）
| 步骤 | finish_reason | 含义 |
|---|---|---|
| 调 list_tables | `tool_calls` | 触发了工具 |
| 调 schema | `tool_calls` | 触发了工具 |
| 调 checker | `tool_calls` | 触发了工具 |
| 调 query | `tool_calls` | 触发了工具 |
| 最终回答 | **`stop`** | 正常完成 |

**前端判断"流式输出是否结束"**：以 `finish_reason == 'stop'` 为准；中间步骤都是 `tool_calls`。

#### 5. Qwen3 prompt cache 实测（重要优化点）
| 步骤 | cached_tokens |
|---|---|
| 1. list_tables | 0 |
| 2. schema | **576** |
| 3. checker | **640** |
| 4. query | **832** |
| 5. 总结 | **896** |

**Qwen3 prompt cache 命中明显**（逐步上升）→ 实施铁律：
- **system_prompt 必须在循环外定义且不变**（已满足）
- **schema 列表的获取**走 `sql_db_list_tables` 后**不重复调用**（避免破坏缓存）
- **同会话多问题**的 token 成本远低于新建会话

#### 6. 端到端耗时与 token 模型
- **单次简单 NL2SQL**：5 次 LLM 往返、~1.3K token、~14s
- **复杂查询（多表 JOIN + 错误重试）**：8-10 次往返、~5K token、~25-35s
- 建议：前端做"AI 思考中"loading 状态，**不能期望秒回**

#### 7. 实施铁律（Phase 3 `p3-nl2sql` 强制遵守）
1. **遍历 `result["messages"]` 找最后一条 `type=='ai'` 的 AIMessage** 作为 final response
2. **`AIMessage.tool_calls[i].args` 直接是 dict**，不要 `json.loads`（这是 LangChain 与 OpenAI SDK 唯一差异点）
3. **`ToolMessage.tool_call_id` 必须回填**为对应 `tool_call.id`
4. **每一步 LLM 调用都记录** `usage_metadata` / `response_metadata`，用于统计
5. **SSE 推送时按 step 增量推 `tool_calls` / `tool_results` / `final`** 三类事件，前端可分别渲染
6. **DB schema 缓存到内存**（避免每次 NL2SQL 都走 schema 工具），但要支持手动刷新
7. **system_prompt 包含完整业务表说明**（提升首次 cache 命中与 SQL 质量）

#### 8. 与 SSE 流式（Phase 4 `p4-sse-stream`）的衔接
- LangChain `agent.stream(stream_mode="values")` 每步推一次完整 messages 列表
- 流式事件映射建议：
  - `messages[-1].type == 'ai' and tool_calls` → push `tool_call` 事件（含 name/args）
  - `messages[-1].type == 'tool'` → push `tool_result` 事件（含 name/content）
  - `messages[-1].type == 'ai' and not tool_calls` → push `final` 事件（含 content）

- **状态**：✅ 已实测落地（2026-06-06），Phase 3 编码强制遵循
- **对应任务**：`p3-nl2sql` / Linear NL2SQL-24（description 已同步）
- **证据脚本**：`backend/test_nl2sql.py`
